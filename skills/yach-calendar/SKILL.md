## 工具：`yach_calendar`（需 OAuth 授权）

### Action 概览

| 意图 | action | 必填参数 | 说明 |
| --- | --- | --- | --- |
| 查日程列表 | `list` | `start_time`, `end_time` | 查询指定时间范围内的日程（支持按工号筛选） |
| 查日程详情 | `detail` | `schedule_id` | 获取单个日程的详细信息 |
| 创建日程 | `create` | `title`, `start_time`, `end_time` | 创建新的日程或会议 |
| 修改日程 | `update` | `schedule_id` | 修改现有日程的信息 |
| 取消日程 | `cancel` | `schedule_id` | 取消已有的日程 |

### 时间格式

**所有时间参数必须使用 ISO 8601 / RFC 3339 格式（含时区）**，例如：
- `'2024-01-01T00:00:00+08:00'` - 北京时间2024年1月1日0点
- `'2024-12-31T23:59:59+08:00'` - 北京时间2024年12月31日23点59分

**注意**：工具内部会将 ISO 格式转换为 Unix 时间戳与后端交互，返回数据中会同时包含 Unix 时间戳和可读的时间字符串（`begin_time_str`, `finish_time_str`）。

## 参数详细说明

### `list` - 查询日程列表

**必填参数**：
- `start_time`：查询开始时间，ISO 8601 格式
- `end_time`：查询结束时间，ISO 8601 格式

**可选参数**：
- `has_cancel`：是否包含已取消日程，`1`=包含 `0`=不包含（默认）
- `has_refuse`：是否包含已拒绝日程，`1`=包含 `0`=不包含（默认）
- `has_self`: 是否包含自己的日程，`1`=包含 `0`=不包含（指定 work_code 时默认为0，否则默认为1）
- `work_code`：工号数组，最多10个，用于筛选指定人员的日程

**注意事项**：
- 时间范围最大不超过 93 天（3 个月）
- `start_time` 必须小于 `end_time`
- 当指定 `work_code` 查询他人日程时，`has_self` 默认为 0（不包含自己），以符合查询他人日程的预期
- 如需在查询他人日程时同时包含自己的日程，需显式设置 `has_self: 1`

**使用示例**：

查询自己日程（默认包含自己）：
```json
{
  "action": "list",
  "start_time": "2024-01-01T00:00:00+08:00",
  "end_time": "2024-01-01T23:59:59+08:00"
}
```

查询他人日程（默认不包含自己）：
```json
{
  "action": "list",
  "start_time": "2024-01-01T00:00:00+08:00",
  "end_time": "2024-01-01T23:59:59+08:00",
  "work_code": ["001", "002"]
}
```

查询他人日程时同时包含自己：
```json
{
  "action": "list",
  "start_time": "2024-01-01T00:00:00+08:00",
  "end_time": "2024-01-01T23:59:59+08:00",
  "work_code": ["001", "002"],
  "has_self": 1
}
```

### `create` - 创建日程

**必填参数**：
- `title`：日程标题
- `start_time`：开始时间，ISO 8601 格式
- `end_time`：结束时间，ISO 8601 格式

**可选参数**：
- `participant`：参与人用户 ID 列表，用英文逗号分隔（**注意：是 UserID，不是工号**）
- `remark`：日程描述，最长 1024 字符
- `address`：会议地点，最长 255 字符（不填默认为线上会议）
- `visibility`：可见性，`1`=公开（默认）`2`=私密
- `is_full`：是否全天日程，`0`=否 `1`=是
- `timezone`：时区，IANA 标识（如 `Asia/Shanghai`），最长 255 字符
- `remind_time`：提醒时间，逗号分隔整数，默认 `'300'` 即 5 分钟前
  - 非全天日程可选值：`-1`=不提醒，`0`=开始时，`180`=3 分钟前，`300`=5 分钟前，`900`=15 分钟前，`1800`=30 分钟前，`3600`=1 小时前，`86400`=1 天前，`172800`=2 天前，`604800`=1 周前
  - 全天日程可选值：`-1`=无提醒，`8`=当天 08:00（默认），`9`=当天 09:00，`10`=当天 10:00，`180`=3 分钟前，`86400`=前 1 天 09:00，`172800`=前 2 天 09:00，`604800`=前 1 周 09:00
- `repeat`：重复类型，`0`=不重复，`1`=按日，`2`=按周，`3`=按月，`4`=按年，`5`=每两周，`6`=工作日，`99`=自定义
- `repeat_end_time`：重复结束时间，ISO 8601 格式，不填或填 `'0'` 表示不限
- `repeat_custom`：自定义重复规则对象（`repeat=99` 时填写），包含：
  - `freq`：重复频率，`1`=按日 `2`=按周 `3`=按月 `4`=按年
  - `interval`：重复间隔（按日/周最大 30，按月/年最大 10）
  - `values`：按日固定填 `'1'`；按周填星期几逗号分隔（`1`=周一…`7`=周日）；按月填日期逗号分隔（如 `'23,24,25'`）
  - `until`：重复次数上限，`0`=不限（结束时间由 `repeat_end_time` 决定）

---

### `update` - 修改日程

**必填参数**：
- `schedule_id`：要修改的日程 ID

**可选参数**：与 `create` 相同（`title`、`start_time`、`end_time`、`participant`、`remark`、`address`、`visibility`、`is_full`、`timezone`、`remind_time`、`repeat`、`repeat_end_time`、`repeat_custom`），但都是可选的

**特殊参数**：
- `scope`：修改范围，`0`=全部（默认）`1`=仅本次 `2`=本次及以后
  - 对于重复日程，**必须先询问用户**要修改的范围再填
  - 单次日程默认使用 `0`

---

### `cancel` - 取消日程

**必填参数**：
- `schedule_id`：要取消的日程 ID

**可选参数**：
- `scope`：取消范围，`1`=取消单次日程或者重复日程的仅本次 `2`=本次及以后
  - 对于重复日程，**必须先询问用户**要取消的范围再填
  - 单次日程默认使用 `1`
- `reason`：取消原因（可选，不传则静默取消）

## 参数组合注意事项

### 0. 智能默认行为
- `has_self` 参数的默认值会根据 `work_code` 参数智能调整：
  - 未指定 `work_code` 时（查询自己日程）：`has_self` 默认为 `1`（包含自己）
  - 指定 `work_code` 时（查询他人日程）：`has_self` 默认为 `0`（不包含自己）
- 这种设计符合用户查询他人日程时的预期，避免意外包含自己的日程

### 1. 时间参数格式
- 所有时间参数必须严格使用 ISO 8601 格式：`YYYY-MM-DDTHH:mm:ss+TZ`
- 错误格式示例：`2024-01-01`、`2024-01-01 00:00:00`
- 正确格式示例：`2024-01-01T00:00:00+08:00`

### 2. UserID 与工号区分
- `participant` 参数使用的是 UserID，不是工号
- 工号格式：如 `167680`
- UserID 格式：通常为数字 ID，如 `34584936`
- 如果用户只提供工号，需要先通过其他工具获取 UserID

### 3. 重复日程的 `scope` 参数
- 重复日程修改/取消时，`scope` 参数特别重要
- `0`=全部：整个重复日程都会被修改/取消
- `1`=仅本次：只修改/取消本次出现的日程
- `2`=本次及以后：从本次开始，后续所有日程都被修改/取消
- **必须先询问用户**要操作的范围

### 4. 重复类型 `repeat` 参数
- `0`=不重复，`1`=按日，`2`=按周，`3`=按月，`4`=按年，`5`=每两周，`6`=工作日，`99`=自定义
- `repeat_end_time`：重复结束时间，ISO 8601 格式，不填或 `'0'` 表示无限重复
- 重复日程创建后，修改/取消时需要特别注意 `scope` 参数

### 5. 提醒时间 `remind_time` 的正确值
- 非全天日程和全天日程的可选值不同
- 非全天日程：`-1`=不提醒，`0`=开始时，`180`=3 分钟前，`300`=5 分钟前，`900`=15 分钟前（默认），`1800`=30 分钟前，`3600`=1 小时前，`86400`=1 天前，`172800`=2 天前，`604800`=1 周前
- 全天日程：`-1`=无提醒，`8`=当天 08:00（默认），`9`=当天 09:00，`10`=当天 10:00，`180`=3 分钟前，`86400`=前 1 天 09:00，`172800`=前 2 天 09:00，`604800`=前 1 周 09:00

## 返回数据字段说明

### 基础字段
- `event_id`：日程唯一 ID，多个重复日程相同
- `schedule_id`：单个日程唯一 ID，修改日程会发生变化
- `title`：日程标题
- `remark`：日程描述，最长 1024 字符
- `address`：日程地点，最长 255 字符；非成员不可见
- `creator`：当前组织者 userid
- `visibility`：可见性，`1`=公开 `2`=私密
- `is_full`：全天日程标识，`0`=非全天 `1`=全天
- `timezone`：时区，IANA 标识（如 Asia/Shanghai）

### 时间字段
- `begin_time`：本次日程开始时间 Unix 秒
- `finish_time`：本次日程结束时间 Unix 秒
- `begin_time_str`：本次日程开始时间可读字符串（工具自动生成）
- `finish_time_str`：本次日程结束时间可读字符串（工具自动生成）
- `created_time`：日程创建时间 Unix 秒
- `repeat_end_time`：重复截止时间 Unix 秒；`0`=无截止
- `util_time`：序列最终截止时间 Unix 秒

### 重复相关字段
- `repeat_value`：重复类型，`0`=不重复 `1`=每日 `2`=每周 `3`=每月 `4`=每年 `5`=每两周 `6`=工作日 `99`=自定义
- `repeat_custom`：自定义重复规则对象；无时为 `{}`
- `is_repeat`：是否重复序列，`0`=单次 `1`=重复序列

### 状态字段
- `state`：日程状态，`0`=创建 `1`=同意 `2`=拒绝 `3`=取消 `4`=修改 `5`=移除 `6`=过期 `99`=删除
- `resp_state`：当前用户响应状态，`0`=待响应 `1`=同意 `2`=拒绝

### 参与人字段
- `participant_num`：参与人总数；非成员返回空字符串
- `resp_accept_num`：已接受人数；非成员返回空字符串
- `participants`：参与人 userid 列表
- `invite_type`：邀请类型，`1`=可邀请 `2`=不可邀请；重复日程强制 `1`

### 其他字段
- `source`：来源，`1`=个人日程 `2`=共享日历 `11`=开放平台 `12`=ICS
- `is_outside`：是否有外部联系人，`0`=否 `1`=是
- `cp_shared`：是否企业共享日程，`0`=个人日程 `1`=企业共享日程
- `is_conflict`：当前时段是否有冲突，`0`=无 `1`=有
- `is_creator`：当前用户是否为本次组织者，`1`=是 `0`=否
- `is_creator_ori`：是否为原始创建者，`1`=是 `0`=否

## OAuth 授权

日历功能需要用户 OAuth 授权，所需的权限：
- `schedule:read` - 查询日程
- `schedule:create` - 创建日程
- `schedule:update` - 修改日程
- `schedule:cancel` - 取消日程

如果用户未授权，会返回需要授权的提示，引导用户完成授权流程。

## 重要改进：智能默认行为

从版本 2.4.0 开始，`yach_calendar` 工具的 `has_self` 参数采用了智能默认值策略：

### 改进动机
- 解决用户查询他人日程时，意外包含自己日程的问题
- 提升用户体验，使工具行为更符合用户预期

### 智能逻辑
```
if (用户指定了 work_code) {
    // 查询他人日程 → 默认不包含自己
    has_self 默认为 0
} else {
    // 查询自己日程 → 默认包含自己
    has_self 默认为 1
}
```

### 开发者建议
- 在大多数情况下，无需显式设置 `has_self` 参数
- 只有在特殊场景下（如查询他人时同时包含自己）才需要显式设置 `has_self: 1`
- 这个改进让 API 调用更加简洁和智能化
